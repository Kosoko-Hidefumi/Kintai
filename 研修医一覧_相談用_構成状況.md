# 研修医一覧：画面切り替えでデータが消える問題 — 相談用構成状況

## 1. 発生している問題

**症状**: 研修医一覧でデータを表示した状態で、他のメニュー（例：期別リスト、修了式資料）に切り替えてから研修医一覧に戻ると、**データが消えてアップロード画面（空状態）になる**。

---

## 2. 技術スタック

| 項目 | 内容 |
|------|------|
| フレームワーク | Streamlit（Python） |
| 研修医一覧の表示 | `resident_dashboard`（React/Vite ビルド） |
| 表示方法 | `st.components.v1.html()` で iframe に HTML を埋め込み |
| データ処理 | `process_data.py` の `process_master_file()`（Excel → JSON 変換） |

## 3. 現在の構成

### 3.1 アプリの動き

```
[メニュー選択]
    ├── カレンダー
    ├── 休暇申請
    ├── 期別リスト
    ├── 修了式資料
    └── 研修医一覧  ← show_resident_dashboard_page()
```

メニュー選択ごとに `main()` が再実行され、選ばれた画面の処理だけが呼ばれます。

### 3.2 研修医一覧（`show_resident_dashboard_page()`）の流れ

**ファイル**: `app.py` 2589行目付近

```python

    st.header("👥 研修医一覧")
    # 1. resident_dashboard/index.html を読み込み
    # 2. assets/*.css, assets/*.js をインラインで HTML に埋め込み

### 3.4 データの流れ（現状）

1. **データの投入経路**  
   - `app.py` 側では、resident_dashboard にデータを渡していない。
   - React 側にファイルアップロードなどがあれば、その画面内でだけデータが保持される想定。

2. **ページ切り替え時**
   - 別メニューを選ぶと `show_resident_dashboard_page()` は呼ばれず、iframe ごと DOM から削除される。
   - iframe 内の React の状態はすべて破棄される。
   - 次に研修医一覧を選ぶと、新しい iframe が作られ、React は初期状態で起動する。
   - 永続化されていないため、以前表示していたデータは復元されない。

3. **localStorage**
   - `app.py` から localStorage に書き込む処理がない。
   - React 側が localStorage を読んでいても、Streamlit の iframe は毎回作り直されるため、origin の扱いによっては期待通り動かない可能性がある。

## 4. 参考：期別リストの実装（データ維持が成功している例）

期別リストでは、以下のようにして画面切り替え後もデータを維持しています。

- **`st.session_state.kibetu_result`**: 処理結果を保持
- **`st.session_state.kibetu_filename`**: ファイル名を保持
- **localStorage**: `kibetu_list_result`, `kibetu_list_filename` に保存
- **HTML コンポーネント**: `kibetu_result` が無い場合は、localStorage から読み取って表示

このため、メニューを切り替えても `session_state` + localStorage でデータが復元されます。

## 7. 制約・前提

1. ** resident_dashboard の React ソースは無い**  
   - minify 済み JS のみ。localStorage の利用や期待するデータ形式はコードからは判別しづらい。
2. **Streamlit の iframe**
   - `st.components.v1.html` は毎回新しい iframe を生成する可能性が高く、origin や sandbox の影響で localStorage の挙動が変わる場合がある。
3. **Streamlit の再実行**
   - メニュー変更のたびに `main()` が再実行され、コンポーネントが再マウントされる。
4. **session_state**
   - 同じ Streamlit セッション内では保持されるため、研修医一覧用のデータもここに持てる。

## 8. 検討してほしいポイント

1. **Streamlit 側でデータを保持する方法**
   - `st.session_state.resident_data` に `process_master_file` の結果を保存
   - 研修医一覧用のファイルアップロード UI を Streamlit 側で実装

2. **iframe 内の React へのデータ渡し**
   - HTML に埋め込む初期化スクリプトで localStorage に書き込む（Base64 などで安全に）
   - または `window.__INITIAL_DATA__` のようなグローバル変数経由（React ソースが無い場合は対応が困難）

3. **localStorage の信頼性**
   - Streamlit の iframe 内の localStorage が期待通り共有・永続するか
   - 親ウィンドウと iframe で origin が同じか

4. **代替案**
   - React を使わず、期別リストと同様に HTML + vanilla JS で PGY カード等を表示する方法
   - または Streamlit のネイティブ UI（`st.dataframe` 等）で表示する方法

## 11. 他AI回答の確認・診断結果（追記）

### 11.1 他AI分析の評価

**結論**: 分析は正確で、推奨方針に同意する。

- **根本原因**: iframe 内の React state のみに依存しているため、メニュー切り替えで iframe が破棄されるとデータが消える。これは仕組み上そのように動く設計。
- **解決の本質**: 「データは Streamlit 側に保持し、画面は毎回そこから復元する」に変えること。
- **二段構えの必要性**: `session_state` だけでは不十分。フロント（React）がデータを受け取る仕組みも必要。

### 11.2 React bundle の localStorage 診断結果

`resident_dashboard/assets/*.js` に対して以下を実行：

- `grep "localStorage"` → **該当なし**
- `grep "sessionStorage"` → **該当なし**
- `grep "getItem"` → **該当なし**
- `grep "resident"` → **該当なし**

**所見**: 現在の resident_dashboard の React バンドルには、**localStorage を利用しているコードが含まれていない**可能性が高い。このため、`window.__RESIDENT_INITIAL_DATA__` や localStorage への注入だけでは、React がそのデータを参照していなければ **表示は復元されない**。

### 11.3 推奨方針の整理

| 優先度 | 方針 | 確実性 | 備考 |
|--------|------|--------|------|
| **1位** | `session_state` を必須基盤とする | 高 | アップロード・クリア・保持を Streamlit 側で実装 |
| **2位** | React が localStorage/`__INITIAL_DATA__` を読むか確認 | 不明 | 診断の結果、読んでいない可能性が高い |
| **3位** | React を捨てて Streamlit/自前 HTML に置き換え | **最も確実** | 期別リストと同様の構成で安定動作が見込める |

### 11.4 実務的な進め方（他AI提案の要約）

1. **フェーズ1**: `st.session_state.resident_data` / `resident_filename`、アップロード UI、`process_master_file()`、クリアボタンを実装（サーバ側の保持を確立）
2. **フェーズ2**: `window.__RESIDENT_INITIAL_DATA__` と localStorage を注入して、React が拾うか検証
3. **フェーズ3**: React が拾わない場合 → **Streamlit/自前 HTML での描画に置き換え**が最短

**一言**: 確実性を重視するなら、**resident_dashboard の置き換え（Streamlit/HTML 描画）** が最確実。

## 13. Streamlit Cloud 対策（2026年3月）

### 13.1 問題

ローカルでは画面切り替え後もデータが保持されるが、**Streamlit Cloud にデプロイすると、メニュー切り替えでデータが消える**。

### 13.2 原因

Streamlit Cloud では、画面切り替え時に `session_state` が保持されない場合がある（Streamlit の既知の挙動。複数ワーカーやリクエスト分散の影響と考えられる）。

### 13.3 実施した対策

**URL のクエリパラメータ `dataid` + サーバー側ストア** で、`session_state` が失われてもデータを復元する。

| タイミング | 処理 |
|------------|------|
| **読み込み時** | `uuid` を生成し、`_RESIDENT_DATA_STORE[uid]` に `(resident_data, resident_filename)` を保存。`st.query_params["dataid"] = uid` で URL に付与。 |
| **表示時** | `session_state` に `resident_data` がなくても、URL に `dataid` があれば `_RESIDENT_DATA_STORE` から復元し、`session_state` に反映。 |
| **クリア時** | ストアから該当データを削除し、URL の `dataid` を除去。 |

### 13.4 実装のポイント

- **`_RESIDENT_DATA_STORE`**: モジュールレベルの `dict`。プロセス生存中は保持される。
- **`dataid`**: URL に含まれるため、Streamlit Cloud 上でも画面遷移後に参照可能。
- **結果**: ローカル・Streamlit Cloud の両方で、画面切り替え後もデータが保持される。

### 13.5 その他の変更

- **`resident_dashboard_data_persistence.md`**: 永続化の検討メモを追加。
- **`.gitignore`**: `resident_dashboard_src/` を追加（他リポジトリのクローンを追跡対象外に）。

## 14. 写真機能の追加可否（検討・未実装）

### 14.1 List-of-Residents の写真機能の仕様

[Kosoko-Hidefumi/List-of-Residents](https://github.com/Kosoko-Hidefumi/List-of-Residents) の resident-dashboard には写真表示機能が含まれている。

| 項目 | 内容 |
|------|------|
| **写真の配置** | `研修医一覧/pictures` または `resident-dashboard/pictures` |
| **ビルド処理** | `npm run build:photos`（`scripts/build-photos.js`）が `pictures` 内の画像を読み込み、ファイル名から「名前」を抽出して `src/data/residentPhotos.json`（名前→Base64 data URL）を生成 |
| **ビルドフロー** | `npm run build` = build-photos → tsc → vite build。写真情報は **ビルド時に** バンドルに埋め込まれる |
| **ファイル名ルール** | `p01山田太郎.jpg`、`01稲村　直紀.jpg` など。先頭の `p`+数字、括弧内は除去。スペース有無・姓名順の違いに対応 |
| **表示場所** | カード表示・テーブル表示・モーダル |
| **依存** | sharp（画像リサイズ・JPEG化）、PIL 相当の処理 |

### 14.2 現行構成への追加可否

**結論: 追加可能。** ただし List-of-Residents の写真は **ビルド時** に埋め込まれる設計のため、運用方法の選択が必要。

### 14.3 運用案の比較

| 案 | 仕組み | メリット | デメリット |
|----|--------|----------|------------|
| **A. ビルド時埋め込み（原設計）** | `pictures/` に写真を置き、`npm run build` で residentPhotos.json を生成しバンドルに含める | 実装済み、オフラインでも表示可能 | 写真の追加・変更のたびに再ビルドが必要。Streamlit Cloud では `pictures/` が git に含まれないためデプロイ時に写真なしになる |
| **B. 実行時注入（Bootstrap）** | app.py 起動時に `kintai/pictures/` を読み、名前→Base64 のマップを作成し、`window.__RESIDENT_PHOTOS__` として Bootstrap で注入 | 写真の追加・変更で再ビルド不要。ローカル運用と相性が良い | React 側の修正が必要（`__RESIDENT_PHOTOS__` を参照）。Streamlit Cloud では `pictures/` がデプロイに含まれないため、Cloud では写真を用意する別手段が必要 |
| **C. ハイブリッド** | ローカル: 実行時注入。Streamlit Cloud: 写真アップロード UI で一時保存し、セッション中だけ表示 | ローカル・Cloud 双方で写真表示を実現可能 | 実装が複雑。Cloud のファイルシステムは再デプロイで消えるため永続化には S3 等が必要 |

### 14.4 推奨方針（要望に沿った形）

ユーザーの方針（「現在の構成を維持」「写真はローカル保存」「次の更新までローカルから読み込み」「写真フォルダは git に含めない」）に合わせる場合：

1. **`kintai/pictures/` を追加し、`.gitignore` に含める**  
   - 写真はローカル専用。git には含めない。

2. **実行時注入（Bootstrap）を採用**  
   - app.py で `kintai/pictures/` を読み、名前→Base64 のマップを生成。
   - Bootstrap で `window.__RESIDENT_PHOTOS__` を渡す。
   - React 側で `__RESIDENT_PHOTOS__` を優先して参照するよう修正。
   - 写真の追加・削除は `pictures/` を編集するだけで反映。再ビルド不要。

3. **Streamlit Cloud の場合**  
   - `pictures/` は git に含めないため、Cloud にデプロイしても写真フォルダは存在しない。
   - Cloud では写真なしで動作させ、写真表示はローカル環境のみ、とする運用が自然。

### 14.5 実装時に必要な変更

| 対象 | 変更内容 |
|------|----------|
| **app.py** | `pictures/` をスキャンし、build-photos.js と同等の名前抽出・リサイズ・Base64 化ロジックを実装。Bootstrap に `window.__RESIDENT_PHOTOS__` を注入 |
| **React（resident_dashboard_src）** | `residentPhotos.json` の代わりに `window.__RESIDENT_PHOTOS__` を優先して参照するように修正。再ビルドして `resident_dashboard/` に反映 |
| **.gitignore** | `pictures/` を追加 |
| **Python 依存** | 画像処理用に Pillow を追加（`requirements.txt`） |

### 14.6 改善提案

1. **`pictures/` の場所を明確にする**  
   - 例: `kintai/resident_dashboard/pictures/` に統一し、app.py から相対パスで参照。README に配置ルールを記載。

2. **Streamlit Cloud で写真を使う場合**  
   - 写真アップロード用の管理者 UI を Streamlit に追加し、アップロード先を S3・Supabase Storage など永続ストレージにすると、Cloud でも写真表示を維持できる。

3. **キャッシュ**  
   - app.py 側で `@st.cache_data` を使って、同じ `pictures/` 内容であれば再スキャン・再エンコードを省略するようにすると、パフォーマンスが向上する。

### 15.2 現在のディレクトリ構成

```

├── app.py                      # メイン。show_resident_dashboard_page() で研修医一覧表示
├── process_data.py             # process_master_file() で Excel → JSON 変換
├── process_photos.py           # pictures/ から画像読み込み、名前→dataURL 辞書を返す

### 15.3 .gitignore で除外しているファイル（pictures 内）

`pictures/` はリポジトリに含めるが、以下の不要ファイルは除外する。

| パターン | 対象 |
|----------|------|

| `pictures/**/.webaxs_*/` | WebDAV キャッシュ |
| `pictures/**/.webview/` | ビューワキャッシュ |
| `pictures/**/._*` | macOS リソースフォーク |
| `pictures/**/.DS_Store` | macOS システムファイル |

### 15.4 データの流れ（写真含む）

```
[Streamlit]
  アップロード → process_master_file() → session_state.resident_data
  pictures/   → process_photos.build_photos_dict() → 名前→dataURL
       ↓
  Bootstrap を <head> に注入
  - window.__RESIDENT_INITIAL_DATA__
  - localStorage（リロード復元用）
  - window.__RESIDENT_PHOTOS__
       ↓
[React]
  useExcelParser → __RESIDENT_INITIAL_DATA__ でデータ取得
  photoMatch → __RESIDENT_PHOTOS__ で写真取得（なければ residentPhotos.json）
       ↓
  マスタ一覧・ダッシュボード・写真付きカードを表示
```
